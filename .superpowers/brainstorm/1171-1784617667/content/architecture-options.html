三种架构方案对比

基于 DeepSeek V4 + 自定义 HTML/JS 前端的技术路线

方案 A：FastAPI + 原生 HTML/CSS/JS（⭐ 推荐）

 ┌─────────────────────────────────────────────────┐
 │              浏览器 (HTML/CSS/JS)                 │
 │  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
 │  │ 聊天面板  │  │ 文件上传  │  │ 模板选择/预览  │  │
 │  └────┬─────┘  └────┬─────┘  └───────┬───────┘  │
 │       └──────────────┼───────────────┘           │
 │                      │ SSE / Fetch                │
 └──────────────────────┼───────────────────────────┘
                        │
 ┌──────────────────────┼───────────────────────────┐
 │               FastAPI (Python)                    │
 │  ┌───────────────────┴──────────────────────┐    │
 │  │              API 路由层                    │    │
 │  │   /chat  /upload  /summarize  /generate   │    │
 │  └────┬──────────────────────────────┬──────┘    │
 │       │                              │           │
 │  ┌────┴──────────┐    ┌──────────────┴──────┐    │
 │  │  Agent 核心    │    │  工具模块            │    │
 │  │  · 对话管理    │◄──►│  · 文档解析器        │    │
 │  │  · 记忆压缩    │    │  · 模板引擎          │    │
 │  │  · 工具调度    │    │  · 搜索客户端        │    │
 │  └───────┬───────┘    └─────────────────────┘    │
 │          │                                       │
 │  ┌───────┴───────┐                               │
 │  │  DeepSeek API  │                               │
 │  └───────────────┘                               │
 └──────────────────────────────────────────────────┘

优势

  • 单一技术栈，部署简单（一个服务全搞定）
  • Python 生态对 LLM/文档处理最友好
  • SSE 原生支持流式输出，聊天体验好
  • 无需 Node.js 构建步骤，直接运行

劣势

  • 纯 JS 前端无组件化，代码组织需自律
  • 复杂 UI 交互需要更多手写代码

方案 B：Flask + Jinja2 + 少量 JS

 ┌──────────────────────────────────────────────────┐
 │              浏览器                               │
 │  页面切换 ──► 全页刷新 / 少量 fetch               │
 └────────────────────┬─────────────────────────────┘
                      │ 传统 HTTP (POST/GET)
 ┌────────────────────┼─────────────────────────────┐
 │            Flask (Python + Jinja2)                │
 │  ┌─────────────────┴──────────────────┐          │
 │  │  路由 → 视图函数 → 渲染模板         │          │
 │  └────────────────────────────────────┘          │
 │  问题：不利于聊天式 UI，大量页面刷新体验差       │
 └──────────────────────────────────────────────────┘

优势

  • 最传统的 Python Web 方案，上手零门槛

劣势

  • 全页刷新体验差，不适合聊天应用
  • SSE/流式输出支持不如 FastAPI

方案 C：FastAPI + Vue/React SPA（过重，不推荐）

 ┌──────────────────┐     ┌──────────────────┐
 │  Vue/React SPA   │────►│  FastAPI 后端     │
 │  (独立开发服务器) │     │  (API 服务器)     │
 │  :5173           │     │  :8000            │
 └──────────────────┘     └──────────────────┘
  需要：Node.js 构建、CORS 配置、双服务部署

优势

  • 组件化开发，大型项目维护性好

劣势

  • 双服务部署复杂，违反"轻量"原则
  • 需要 Node.js + Python 双环境
A

FastAPI + 原生 HTML/CSS/JS

轻量、高效、一流聊天体验。最符合项目"轻量"定位
