三种架构方案对比

基于 DeepSeek V4 + 自定义 HTML/JS 前端的技术路线

方案 A：FastAPI + 原生 HTML/CSS/JS ⭐ 推荐

🌐 浏览器 — 单页应用 (HTML/CSS/JS)
💬 聊天面板 📎 文件上传 📋 模板预览
⬇️ SSE 流式 / Fetch 请求
⚡ FastAPI 服务器 (Python)
🔀 API 路由层
/chat /upload /summarize /generate /search
🧠 Agent 核心
对话管理器
上下文记忆（摘要压缩）
工具调度器（Function Calling）
🔧 工具模块
📄 文档解析器（txt/docx/md/pdf）
📝 模板引擎（Jinja2）
🔍 搜索客户端
⬇️ HTTPS + API Key
☁️ DeepSeek V4 API （兼容 OpenAI SDK）

优势

  • 单一 Python 服务，python app.py 即可运行
  • SSE 原生支持流式输出，打字机效果体验好
  • DeepSeek SDK、文档库全部 Python 原生
  • 无需 Node.js，零构建步骤

劣势

  • 纯 JS 无组件化，代码组织需自律
  • 复杂 UI 交互需手写较多代码

方案 B：Flask + Jinja2 模板 + 少量 JS

🌐 浏览器
全页刷新为主 / 少量 fetch
⬇️ 传统 POST/GET
🐍 Flask + Jinja2 渲染
路由 → 视图 → 模板 → 全页返回

优势

  • 最简单，上手零门槛

劣势

  • 全页刷新体验差，不适合聊天应用
  • SSE 流式输出支持远不如 FastAPI

方案 C：FastAPI + Vue/React SPA（过重）

⚛️ React/Vue SPA
:5173 dev server
⇄
⚡ FastAPI
:8000 API

优势

  • 组件化开发，大型项目维护性好

劣势

  • 双服务部署复杂，违反"轻量"原则
  • 需要 Node.js + Python 双环境
A

FastAPI + 原生 HTML/CSS/JS

轻量、高效、一流聊天体验。最符合项目"轻量"定位
